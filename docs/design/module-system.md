# Module System Architecture

## Overview

The Module System is the primary abstraction for computation within the Generic Execution Framework (GEF). It provides a unified model for defining, composing, and executing algorithmic units. In GEF, a **Module** is the fundamental building block of any application, ranging from a single image processing operation to a complex, multi-threaded video analysis pipeline.

The architecture is built on three core pillars:
1.  **Unified Abstraction**: Every unit of computation, whether a single C++ function or a nested DAG of other modules, shares the same interface contract.
2.  **Explicit Dataflow**: Modules communicate exclusively through declared bindings (Inputs, Outputs, InOuts, and Configs), enabling the framework to automate scheduling and resource management.
3.  **Recursive Composition**: The system utilizes the Composite Pattern, allowing modules to be nested indefinitely to manage complexity and promote reuse.

By standardizing how computation is declared and connected, the Module System enables the GEF scheduler to perform static analysis, validate data lineage, and maximize hardware utilization through automatic parallelization.

## Composite Pattern

GEF strictly adheres to the **Composite Pattern**. From the perspective of the framework, there is no structural difference between an Atomic Module (implemented in C++) and a Composite Module (composed of other modules via YAML).

Every module type exposes the same mandatory interface:
-   **Inputs**: A set of required data resources produced by upstream modules.
-   **Outputs**: A set of new data resources produced by this module.
-   **Configs**: A set of immutable parameters injected by the system.

This uniformity allows a `Pipeline` to contain a `Flow`, which in turn contains a `Batch` wrapper around an `Atomic` module. To the parent module or the top-level `System`, the internal complexity of a child module is encapsulated behind its interface contract.

## Atomic Module

An **Atomic Module** is the leaf node of the execution graph. It represents a single, indivisible unit of algorithm logic implemented in C++.

### Implementation Model
Atomic modules are designed for a "researcher-first" development experience. Each module is typically defined in a single `.cpp` file and compiled into a standalone shared library (`.so` on Linux, `.dylib` on macOS). This plugin-based architecture allows for rapid iteration: a researcher can modify a single file, rebuild only that library, and the `System` can immediately load and execute the updated logic without a full application relink.

The framework discovers these modules by scanning designated plugin directories. Each library must export a metadata query function (automatically generated by the macros) that allows the system to inspect the module's requirements before loading the heavy executable code.

### Declaration Interface
Atomic modules are declared using a set of C++ macros that handle the registration and metadata generation required by the framework. These macros create the necessary glue code to bridge the researcher's logic with the engine's internal task runner.

-   `GEF_STAGE(Name)`: Defines the entry point for the module. Note that while the macro uses "STAGE" for historical compatibility and registration reasons, the component is strictly referred to in all documentation and system logic as an Atomic Module.
-   `GEF_INPUT(Type, name)`: Declares a read-only input. This informs the scheduler that the module requires this data to be ready before it can start.
-   `GEF_OUTPUT(Type, name)`: Declares a newly produced output. The framework ensures a fresh resource is allocated for this binding.
-   `GEF_INOUT(Type, name)`: Declares an in-place mutation of existing data. This is a high-performance binding that requires exclusive access to the resource.
-   `GEF_CONFIG(Type, name)`: Declares an immutable configuration parameter, usually sourced from a global YAML file.

### Execution Logic and Context API
The primary responsibility of the researcher is to implement the `execute(Context& ctx)` method. The `Context` object is the sole gateway to the framework's data and services during execution. It provides type-safe access to the declared bindings via proxy types (like `InputRef<T>`).

These proxy types are a critical design feature: they prevent accidental copies of large data structures (like 4K video frames) when researchers use the `auto` keyword. The proxy behaves like a reference but is managed by the framework to ensure safety and instrumentation.

```cpp
GEF_STAGE(GrayscaleFilter) {
    GEF_INPUT(Image, input_image);
    GEF_OUTPUT(Image, gray_image);
    GEF_CONFIG(bool, use_high_precision);

    void execute(Context& ctx) {
        // Accessing data through the Context API
        auto in = ctx.input<Image>("input_image");
        auto out = ctx.output<Image>("gray_image");
        auto precision = ctx.config<bool>("use_high_precision");

        // The framework ensures 'in' is valid and 'out' is allocated
        // before this method is called.
    }
};
```

## Pipeline

A **Pipeline** is a composite module that executes its child modules in a strict, sequential order. It is the simplest form of composition, ideal for linear workflows where each step depends on the result of the previous one.

### Semantics
Pipelines use **variable list semantics**. As data flows through the pipeline, each module adds its outputs to the available scope or modifies existing data via InOut. This creates a "sliding window" of data availability where the state of the system is evolved step-by-step.

-   **Implicit Wiring**: By default, the framework matches module inputs to upstream outputs by name. If Module B requires an input named "image", it will automatically consume the output named "image" produced by Module A.
-   **Skip Connections**: A module's output is available to all subsequent modules in the pipeline, not just the immediate successor. For example, if module A produces `tensor_1`, module C can consume it even if module B does not. This mimics how researchers often think about "scripts" where variables persist until the end of the execution.
-   **Last-Writer-Wins**: If multiple modules produce an output with the same name, the most recent output supersedes previous ones for downstream consumers. While this allows for "overwriting" semantics, the validation layer issues a warning when this name collision occurs to ensure the researcher is aware of the shadowing effect.

### Example Configuration
```yaml
module:
  name: image_prep_pipeline
  type: pipeline
  modules:
    - name: reader
      type: image_reader
    - name: blur
      type: gaussian_blur
    - name: threshold
      type: adaptive_threshold
```

## Flow

A **Flow** is a composite module that defines a Directed Acyclic Graph (DAG) of child modules. Unlike a Pipeline, a Flow allows for explicit parallel execution of modules that do not share data dependencies.

### Semantics
Flows prioritize clarity and performance over implicit convenience. They are used when the researcher wants to take advantage of multi-core CPUs or heterogeneous compute resources by running independent branches of the algorithm simultaneously.

-   **Explicit Wiring**: Connections between modules are explicitly defined in the YAML configuration. This can be done using a concise arrow syntax or a more structured format.
    -   Arrow syntax: `blur.output -> detect.input`
    -   Structured: `{ from: blur.output, to: detect.input }`
-   **Parallel Execution**: The scheduler performs a topological sort on the Flow's DAG and groups modules into parallel execution buckets. Modules with no mutual dependencies (neither direct nor indirect through `InOut` resources) are executed concurrently across available hardware threads.
-   **Collision Safety**: Unlike the Pipeline's last-writer-wins rule, a name collision in a Flow is a **Build Error** unless the ambiguity is resolved through explicit wiring. This prevents subtle race conditions where two parallel branches might otherwise attempt to write to the same named output.

### Scheduling Rationale
By requiring explicit wiring in Flows, GEF ensures that the dataflow is deterministic. The scheduler uses this information to optimize memory reuse—for instance, if it knows two modules in different branches don't overlap in time, it can potentially reuse the same memory buffer for their outputs.

## Batch

A **Batch** is a wrapper module that enables data-parallel execution. It takes a single inner module (which can be Atomic, Pipeline, or Flow) and executes it multiple times across a collection of data items. This is particularly powerful for video processing, where each frame can be processed independently.

### Transparency and Encapsulation
The inner module is **transparent** to the batching process. It is written as if it were processing a single item. This separation of concerns means that a researcher can develop a complex single-frame enhancement pipeline and then "batchify" it simply by wrapping it in a Batch module in the YAML configuration.

### Split and Gather Strategies
The Batch module employs specific strategies to handle the "Scatter/Gather" pattern:

-   **Split**: Determines how the input resource is divided into individual tasks.
    -   `batch_tensor_dim`: Slices a large tensor along a specified dimension. For example, splitting a `[100, 3, 224, 224]` tensor into 100 individual `[3, 224, 224]` image tensors.
    -   `batch_sequence`: Iterates through a `List<T>` or similar collection, producing one task per element.
    -   `custom`: Allows the researcher to provide a specialized module that performs complex partitioning (e.g., splitting an image into overlapping patches).

-   **Gather**: Determines how individual outputs are recombined into a single result resource.
    -   Strategies typically mirror the split strategy (e.g., `batch_tensor_dim` gather concatenates slices back into a single tensor).
    -   The Gather strategy is also responsible for error handling—if one element in the batch fails, the Gather strategy defines whether the whole batch fails or if the partial results are returned with a warning.

### Race Detection on In-Place Mutation
A unique challenge of the Batch module is managing `InOut` bindings. If a Batch module is performing in-place mutation on a shared resource, the framework must ensure that no two parallel iterations are writing to the same memory region.
-   **Static Detection**: The scheduler checks if the split strategy guarantees non-overlapping regions.
-   **Runtime Protection**: If static analysis is insufficient, the framework uses runtime checks (in Debug mode) to detect overlapping memory access, preventing silent data corruption.

## Branch

**Branching** is categorized as a system-level capability rather than a standard module type. It handles runtime routing logic (e.g., `if/else` or `switch` patterns) where the actual execution path depends on data values computed during the run.

### The Role of the Dynamic Scheduler
In GEF, the standard Module System is "Static"—the structure of Pipelines and Flows is known before execution begins. Branching introduces "Dynamic" behavior. This is handled by a specialized **Dynamic Scheduler** that evaluates conditions at runtime and then triggers the execution of the appropriate static sub-graph.

For example, a video analysis system might use a Branch to decide whether to run a "High-Complexity Tracker" or a "Low-Power Tracker" based on the remaining battery level or the number of objects detected in the current frame. This design preserves the simplicity and performance of the static module graph while providing the necessary flexibility for complex real-world applications.

## Nesting Rules and Hierarchical Composition

The Module System is recursively defined, adhering to the principle that a composite is also a module. This allows for arbitrary levels of nesting to model hierarchical logic.

### Supported Nesting Patterns
1.  **Pipeline within a Flow**: A parallel branch in a DAG can be implemented as a sequential pipeline of operations.
2.  **Flow within a Pipeline**: A step in a sequential pipeline can be a complex DAG that utilizes multiple cores for a specific sub-task.
3.  **Batch wrapping a Pipeline**: A sequential sequence of steps can be applied in parallel to every item in a collection.
4.  **Batch wrapping a Flow**: An entire parallel processing graph can be executed for every frame of a video.

### Composition Limits
While there is no technical limit to nesting depth, the framework encourages researchers to use composition to manage complexity. A "flat" graph with 100 modules is often harder to debug than a hierarchical graph where those 100 modules are grouped into 10 logical sub-flows.

## Interface Contract and Validation

Every module, regardless of its internal implementation or nesting level, must fulfill a strict **Interface Contract**. This contract is the "Legal Agreement" between the module and the framework.

### Mandatory Declarations
The contract requires the explicit declaration of:
-   **Inputs**: The set of data resources (name and type) that the module expects to receive.
-   **Outputs**: The set of data resources (name and type) that the module promises to produce.
-   **Configs**: The parameters (name and type) used to tune the module's behavior.

### Validation Phases
The GEF validation layer uses these contracts to perform checks at different stages of the development lifecycle:
1.  **Build Phase**: Parses YAML and C++ metadata to ensure that all modules in a graph are correctly wired. It detects "Unresolved Bindings" (input requested but not produced) and "Type Mismatches".
2.  **Schedule Phase**: Analyzes the dependency graph to find "InOut Races" where two parallel modules might attempt to mutate the same resource.
3.  **Runtime Phase (Debug)**: Verifies that modules actually produce the outputs they promised and checks for "InOut No-Write" (where a researcher declared an InOut but only performed reads).

## Design Rationale: Why a Unified Module Abstraction?

The decision to treat every unit of work as a "Module" with a consistent interface was driven by the core GEF philosophy of **Explicit over Implicit**.

### 1. Unified Tooling
Because every module (Atomic or Composite) looks the same from the outside, we can build a single set of tools for profiling, debugging, and visualization. A profiler doesn't need to know if a module is a simple C++ function or a nested DAG; it just tracks the time between the start and end of the `execute` call.

### 2. Researcher vs. Engineer Separation
The module system defines a clear boundary between the **Researcher** (who writes the algorithmic modules) and the **Engineer** (who composes them into a production system). Researchers focus on the C++ logic within an Atomic Module, while Engineers focus on the YAML composition, resource pooling, and scheduling strategies.

### 3. Safety without Compromise
The explicit contract allows GEF to provide safety guarantees (like race detection and type checking) that are usually only found in much more restrictive frameworks. By making these checks part of the framework's "Contract Negotiation," we provide safety without imposing a significant performance penalty on the actual computation.

### 4. Composability and Reuse
The unified abstraction makes every module a "Black Box" that can be dropped into any other Pipeline or Flow. This promotes a culture of reuse, as a well-written Atomic module or a highly optimized Flow can be shared across multiple projects with no changes to its internal code.

## Relationship to Other Components

The Module System is the central "Structure" upon which the rest of the GEF engine is built. Its relationships with other subsystems are clearly defined:

-   **Scheduler**: The scheduler is the "Director" that reads the module contracts and the YAML wiring to decide the order of operations. It is responsible for fulfilling the requirements of the module system's DAG.
-   **ResourcePool**: The ResourcePool is the "Warehouse" of memory. When a module's contract says it produces an output, the ResourcePool provides the actual memory buffer. The Module System defines the *demand*, while the ResourcePool handles the *supply*.
-   **Validation Layer**: The validation layer is the "Inspector" that ensures all module contracts are being honored. It works closely with the module system to catch errors as early as possible.
-   **System**: The System is the "Executive" that owns everything. it initializes the ResourcePools, loads the Module plugins, and hands the final graph to the Scheduler for execution.
