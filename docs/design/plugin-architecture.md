# Plugin Architecture

## Overview

The Generic Engine Framework (GEF) utilizes a decentralized plugin-based architecture for its core computation units. In GEF, every Atomic Module is treated as an independent plugin, physically residing as a shared library (`.so` on Linux, `.dylib` on macOS, or `.dll` on Windows).

This model moves away from the traditional monolithic framework approach where modules are statically linked into the main executable. Instead, the GEF engine acts as a host that discovers, loads, and orchestrates these external units of logic. By decoupling the algorithm implementation from the engine core, GEF achieves high flexibility and a "researcher-first" development experience.

## Build Model

The plugin architecture is specifically designed to minimize build times and maximize researcher productivity. The fundamental unit of work is a single `.cpp` file.

1.  **Isolation**: Each Atomic Module is a standalone compilation unit.
2.  **Granularity**: Modifying a module's algorithm only requires recompiling that specific `.cpp` file into its corresponding shared library.
3.  **Hot-Reload Capability**: Because the engine loads these libraries dynamically, a researcher can modify a module, run a single compiler command, and the updated logic is immediately available for the next execution run without needing to relink the entire application.

This "edit-rebuild-run" cycle is reduced from minutes (in large monolithic projects) to seconds, enabling a rapid experimentation flow similar to interpreted languages while maintaining C++ performance.

## Metadata Exposure: The Query Function Pattern

For the engine to schedule and wire modules correctly, it must know their interface (Inputs, Outputs, InOuts, and Configs) before actually executing them. GEF uses a **Query Function Pattern** to expose this metadata.

Each plugin must export a single C-linkage function:
```c
extern "C" const gef_metadata_t* gef_get_metadata();
```

This function returns a pointer to a static metadata structure containing:
-   **Module Name**: The unique identifier for the module type.
-   **Binding Declarations**: A list of names, types, and roles (Input, Output, InOut, Config).
-   **API Version**: The version of the GEF API the module was compiled against.

The engine calls this function immediately after loading the library but before registering it. This allows the framework to validate the module's compatibility and requirements without running any algorithmic code.

## What Macros Generate

Researchers define modules using high-level macros like `GEF_STAGE`. These macros are not just syntactic sugar; they generate the boilerplate required for the query function pattern.

### User Code (Before Expansion)
```cpp
GEF_STAGE(GrayscaleFilter) {
    GEF_INPUT(Image, input_image);
    GEF_OUTPUT(Image, gray_image);
    GEF_CONFIG(float, threshold);

    void execute(Context& ctx) {
        // Algorithm logic here
    }
};
```

### Generated Code (Conceptual Expansion)
```cpp
// Hidden implementation details generated by macros
class GrayscaleFilter : public AtomicModule {
public:
    void execute(Context& ctx) override;
};

static const gef_binding_t s_bindings[] = {
    {"input_image", "Image", GEF_ROLE_INPUT},
    {"gray_image", "Image", GEF_ROLE_OUTPUT},
    {"threshold", "float", GEF_ROLE_CONFIG}
};

static const gef_metadata_t s_metadata = {
    .version = GEF_API_VERSION,
    .name = "GrayscaleFilter",
    .bindings = s_bindings,
    .binding_count = 3,
    .create_func = []() -> AtomicModule* { return new GrayscaleFilter(); }
};

extern "C" const gef_metadata_t* gef_get_metadata() {
    return &s_metadata;
}
```

## Module Discovery

At startup, the `System` performs a discovery phase to locate available modules.

1.  **Directory Scanning**: The engine scans a set of configurable search paths (e.g., `./plugins`, `/usr/local/lib/gef`).
2.  **File Filtering**: It looks for files with platform-specific shared library extensions.
3.  **Lazy Inspection**: The engine performs a shallow load of each library to check for the presence of the `gef_get_metadata` symbol. Libraries missing this symbol are ignored.
4.  **Registration**: Successfully inspected modules are added to the internal registry, indexed by their metadata name.

## Loading Protocol

The lifecycle of a plugin follows a strict protocol:
1.  **Discovery**: Identify potential `.so`/`.dylib` files.
2.  **Dynamic Loading**: Use `dlopen` (or `LoadLibrary`) to bring the library into the process address space.
3.  **Query**: Invoke `gef_get_metadata()` to retrieve the interface contract.
4.  **Validation**: Verify that the module's API version is compatible with the engine.
5.  **Type Registration**: Store the module's metadata and creation factory in the global registry.
6.  **Instantiation**: When a YAML config references the module by name, the engine uses the factory to create a runtime instance.
7.  **Execution**: The scheduler invokes the module's `execute()` method as part of the graph execution.

## API Versioning

To prevent crashes caused by ABI mismatches, the `gef_metadata_t` struct includes a version field.
-   **Backward Compatibility**: The engine checks the module's version against its own supported range.
-   **Evolution**: As the metadata structure evolves (e.g., adding new binding roles or execution hints), the version number is incremented.
-   **Safety**: If a module is compiled against an incompatible GEF header, the engine refuses to load it, providing a clear error message instead of a segmentation fault.

## Cross-Platform Considerations

GEF abstracts the platform-specific details of dynamic loading.
-   **Linux**: Uses `dlopen`/`dlsym` with `.so` extensions.
-   **macOS**: Uses `dlopen`/`dlsym` with `.dylib` extensions.
-   **Windows**: Uses `LoadLibrary`/`GetProcAddress` with `.dll` extensions.

The source code for an Atomic Module remains identical across platforms. The GEF build system handles the differences in compiler flags and linker settings to produce the correct plugin format for the host OS.

## Why Query Function Over Binary Sections

Some systems (like the Linux kernel or certain ELF-based frameworks) use custom binary sections (e.g., `.note.gef`) to store metadata. GEF intentionally chose the `extern "C"` query function approach for several reasons:

1.  **Simplicity**: Implementing a query function requires no special linker scripts or platform-specific object file manipulation.
2.  **Cross-Platform Portability**: `extern "C"` functions are a standard part of the C/C++ ABI across all major operating systems. Binary sections are highly platform-dependent (ELF vs. Mach-O vs. PE).
3.  **Debuggability**: Metadata can be easily inspected using standard tools like `nm` or `objdump`. Developers can even write simple C programs to load a plugin and print its metadata for verification.
4.  **Efficiency**: Reading a static struct via a function call is extremely fast and avoids the overhead of parsing complex binary headers.

## Security Considerations

Loading and executing third-party shared libraries carries inherent risks.
-   **Trust Model**: GEF operates on a trusted-plugin model. It assumes that plugins in the designated search paths are authorized by the user.
-   **Address Space**: Plugins run in the same address space as the host engine. A crash in a plugin will crash the entire system.
-   **Sandboxing**: While not part of the core GEF implementation, the plugin architecture allows for future extensions where specific modules could be run in isolated processes or containers if security requirements dictate.

## Relationship to YAML Config

The YAML configuration files bridge the gap between the physical plugins and the logical execution graph.

-   **Name Matching**: When a YAML file specifies a module `type: GrayscaleFilter`, the engine looks up "GrayscaleFilter" in its registry of loaded plugins.
-   **Binding Verification**: The engine uses the plugin's metadata to ensure that the wiring provided in the YAML (inputs and outputs) matches the module's actual requirements.
-   **Parameter Injection**: The `configs` declared in the metadata are used to validate and type-check the values provided in the YAML configuration before execution begins.
